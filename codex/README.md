# Codex (OpenAI Codex CLI)

作成日: `2026-05-03`

## 1. 概要

Codex CLI は OpenAI が提供するターミナルベースのコーディングエージェントです。
テキスト生成・設計整理・ドキュメント作成・実装委譲に強く、リーダー業務の補助に向きます。

このスタックでは **リーダー業務と実装委譲の第一候補** として位置づけます。

---

## 2. 主な特徴

| 特徴 | 内容 |
| --- | --- |
| リーダー業務 | ヒアリング整理、設計分解、WBS、見積もり、報告文書作成 |
| 実装委譲 | タスク分解して実装・テスト・PR 作成まで行える |
| コードレビュー | 差分確認、改善案提示 |
| ドキュメント | README、設計メモ、運用手順、比較表作成 |
| 承認モード | 危険な操作は都度確認するモードと自動実行モードを選べる |

### 得意な業務

- ヒアリング項目の作成・議事録整理・決定事項・未決事項の構造化
- 基本設計のたたき台作成・複数案比較
- WBS 作成・依存関係整理
- 見積もり補助（タスク分解・不確実性整理）
- 報告文書（上長向け・顧客向け・開発者向け）
- README・運用手順・設計メモ作成
- 実装タスクの分解と委譲

```bash
# プロジェクトディレクトリに移動

codex
```

## 5. AGENTS.md（プロジェクト指示ファイル）

プロジェクトルートに `AGENTS.md` を置くと、Codex が起動時に自動で読み込みます。

### 役割

- プロジェクト固有のルール・制約・禁止事項を記述する
- 使用技術スタック・ディレクトリ構成・命名規則を伝える
- 実行してよいコマンド・してはいけない操作を明記する

### テンプレート

```markdown
# AGENTS.md

## プロジェクト概要
- 用途: ...
- 主言語: Python / Rust
- DB: PostgreSQL

## 禁止事項
- `terraform apply` は人間承認必須
- 本番 DB への直接 UPDATE / DELETE 禁止
- `main` ブランチへの直接 push 禁止

## 技術スタック
- backend: Python 3.11 / Rust 1.75
- DB: PostgreSQL 15
- IaC: Terraform 1.6

## ディレクトリ構成
- src/: メインコード
- tests/: テストコード
- infra/: Terraform

## よく使うコマンド
- テスト実行: `pytest tests/`
- Rust ビルド: `cargo build`
- DB migration: `alembic upgrade head`

## コミットメッセージ規則
```

## 5-1. Rules（最初に押さえるべき基本機能）

Codex で品質を安定させるには、最初に Rule を固定します。

### Rule の優先順位

| レイヤー | 例 | 使い方 |
| --- | --- | --- |
| セッション指示 | 対話で渡す依頼文 | その作業だけの一時指示 |
| リポジトリ指示 | `AGENTS.md` | プロジェクト共通の標準ルール |
| 個人設定 | `~/.codex/config.*` | 全プロジェクト共通の個人設定 |

### 最小 Rule セット（必須）

- 破壊的操作の禁止: `terraform apply`、本番 DB write、`main` 直 push
- テスト実行の必須化: 変更後は必ずテストを実行
- 変更単位の制約: 1 PR 1 目的、巨大差分を避ける
- セキュリティ制約: 秘密情報をログや PR に出さない

### Rule 追記例

```markdown
## 品質ルール
- 変更後は `pytest tests/` を実行し、失敗時は原因と対策を記録する
- 既存 API の互換性を壊す変更は事前に明示する

## セキュリティルール
- `.env`、鍵ファイル、トークンを出力しない
- 本番環境に対する write 操作は人間承認必須
```

## 5-2. Skills（再利用できる手順の基本機能）

Skills は、繰り返し作業を定型化するための実務テンプレートです。
毎回プロンプトをゼロから書かずに、手順と完了条件を再利用できます。

### Skill 化すべき代表パターン

- 要件分解（機能要件/非機能要件/制約/未決事項）
- 見積もり（WBS 化、依存関係、リスク明示）
- 実装委譲（対象ファイル、完了条件、テスト条件）
- PR レビュー（重大/中/軽微分類、テスト不足抽出）

### Skill テンプレート（例）

```markdown
# Skill: PR レビュー

## 入力
- 変更差分
- 関連テスト

## 手順
1. 破壊的変更と後方互換性を確認
2. 例外処理と入力検証を確認
3. テスト不足を抽出
4. 重大/中/軽微で指摘を分類

## 完了条件
- 再現可能な根拠付きで指摘を列挙
- 必須修正と任意改善を分離
```

## 5-3. Agents（役割分担の基本機能）

Agents は、タスクを役割ごとに分担して品質を安定させる仕組みです。

### 最小 Agent 構成（推奨）

| Agent | 役割 | 禁止事項 |
| --- | --- | --- |
| 実装 Agent | コード修正・テスト追加 | 本番環境への直接変更 |
| レビュー Agent | 差分監査・回帰リスク洗い出し | 実装修正の直接実行 |
| ドキュメント Agent | README、変更記録、運用手順更新 | コードの仕様変更 |

### Agent 指示の書き方

- 目的: 何を達成するか
- 入力: 参照ファイル、ログ、関連仕様
- 完了条件: テスト結果、差分条件、報告形式
- 禁止事項: 本番操作、機密情報出力、巨大差分

### Agent 指示テンプレート

```markdown
## Agent: 実装

目的:
- 指定した不具合を最小差分で修正する

完了条件:
- 再現手順で不具合が解消
- 追加/既存テストが通過
- 変更点と影響範囲を3行で報告
```

## 6. 承認モード

| モード | 説明 | 推奨場面 |
| --- | --- | --- |
| `suggest`（デフォルト） | 提案のみ、実行は人間が承認 | 本番・重要なプロジェクト |
| `auto-edit` | ファイル編集は自動、コマンド実行は確認 | 開発環境での実装作業 |
| `full-auto` | すべて自動実行 | 安全なサンドボックス環境のみ |

```bash
# 承認モードを指定して起動
codex --approval-mode auto-edit "テストコードを追加してください"
```

## 8. 運用のポイント

| ポイント | 内容 |
| --- | --- |
| AGENTS.md を必ず用意する | プロジェクトのルール・禁止事項を事前に記述する |
| 見積もりは参考値として扱う | AI の出力をそのまま採用しない |
| Cloud / DB 操作は権限制限 | 本番環境への write 権限は渡さない |
| 大きな変更は PR 単位に分割 | 一度に大量の変更をさせない |
| full-auto は安全環境のみ | 本番・重要環境では `suggest` または `auto-edit` を使う |
