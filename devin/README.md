# Devin

作成日: `2026-05-03`

## 1. 概要

Devin は Cognition AI が開発した完全自律型の AI ソフトウェアエンジニアです。
ブラウザ・ターミナル・エディタを組み合わせて、チケット単位のタスクを自律的に実行します。

このスタックでは **チケット単位で丸ごと委譲する用途** として位置づけます。

---

## 2. 主な特徴

| 特徴 | 内容 |
| --- | --- |
| 完全自律実行 | ブラウザ・ターミナル・エディタを自律的に操作する |
| チケット委譲 | 明確な完了条件があるタスクを丸ごと任せられる |
| ブラウザ検証 | Web UI の動作確認・スクリーンショット取得ができる |
| PR 作成 | 実装後に GitHub PR を自動で作成できる |
| 並行作業 | 複数のタスクを並行して進められる |
| Playbook | 繰り返すタスクの手順を Playbook として保存・再利用できる |

### 得意な業務

- 小さな機能追加（明確な完了条件があるもの）
- バグ再現と修正
- テスト追加
- 既存コードの棚卸し・リファクタ
- 移行作業（ライブラリ更新、API 変更への追従）
- ブラウザ検証込みのタスク
- バックログ処理（定型的・繰り返し可能なタスク）

---

## 3. 環境構築

### アクセス方法

Devin は Web ベースのサービスです。ローカルへのインストールは不要です。

1. [app.devin.ai](https://app.devin.ai) にアクセス
2. GitHub アカウントでサインイン
3. リポジトリへのアクセス権限を付与

### GitHub 連携設定

1. `Settings > Integrations > GitHub` でリポジトリを接続
2. Devin が PR を作成できるよう権限を設定

### Slack 連携（オプション）

Slack ワークスペースと連携すると、Slack から直接タスクを依頼できます。

## 4. 起動・使用方法

### 基本的なタスク依頼

```
# Web UI でタスクを入力する例

タスク: ユーザー一覧APIにページネーションを追加してください

完了条件:
- GET /api/users?page=1&per_page=20 で動作すること
- テストが通ること
- PR が作成されること

禁止事項:
- 本番 DB への直接操作禁止
- main ブランチへの直接 push 禁止
```

### 効果的なタスク記述のポイント

```markdown
## タスクテンプレート

### 目的
[何を達成したいか]

### 完了条件
- [ ] [具体的な成果物・動作の基準]
- [ ] テストが通ること
- [ ] PR が作成されること

### 技術的な前提
- 言語: Python 3.11
- フレームワーク: FastAPI
- DB: PostgreSQL 15

### 禁止事項
- 本番 DB への直接操作禁止
- main ブランチへの直接 push 禁止
- [プロジェクト固有の禁止事項]

### 参考情報
- 関連ファイル: src/api/users.py
- 既存テスト: tests/test_users.py
```

## 5. Playbook（繰り返しタスクの自動化）

Playbook は繰り返し実行するタスクの手順書です。

### Playbook の作成例

```markdown
# Playbook: 新機能追加の標準手順

## ステップ

1. 既存コードを読んで影響範囲を確認する
2. ブランチを切る: `git checkout -b feature/[機能名]`
3. 実装する
4. テストを追加する: `pytest tests/` が通ること
5. リントを実行する: `ruff check src/`
6. PR を作成する（タイトル・本文テンプレートを使う）

## PR テンプレート

### 変更内容
[変更の概要]

### 影響範囲
[変更が影響するファイル・機能]

### テスト
- [ ] ユニットテスト追加
- [ ] 既存テストが通ること確認
```

## 6. Knowledge（プロジェクト固有の知識）

Devin の Knowledge に以下を登録しておくと作業品質が上がります。

```markdown
# プロジェクト Knowledge

## 禁止事項
- Next.js は使用禁止
- terraform apply は人間承認必須
- 本番 DB への直接 write 禁止

## 技術スタック
- backend: Python 3.11 / Rust 1.75
- DB: PostgreSQL 15
- frontend: React + pico.css

## よく使うコマンド
- テスト: `pytest tests/ -v`
- リント: `ruff check src/`
- 型チェック: `mypy src/`
- DB migration: `alembic upgrade head`

## ブランチ・PR 規則
- ブランチ名: `feature/[機能名]`、`fix/[バグ名]`
- PR タイトル: `feat: ...`、`fix: ...`、`docs: ...`
- PR はドラフトで作成してから人間がレビュー
```

## 6-1. Rules（最初に押さえるべき基本機能）

Devin は自律実行のため、Rule を先に固定しないと事故率が上がります。
タスク本文と Knowledge に必須ルールを明示してください。

### Rule の優先順位

| レイヤー | 例 | 使い方 |
| --- | --- | --- |
| タスク指示 | チケット本文の目的・完了条件・禁止事項 | そのタスクだけの最優先指示 |
| プロジェクト知識 | Devin Knowledge | リポジトリ共通ルール |
| 組織ポリシー | GitHub / Cloud 権限制約 | 環境全体の強制ルール |

### 最小 Rule セット（必須）

- 破壊的操作の禁止: `terraform apply`、本番 DB write、`main` 直 push
- テスト実行の必須化: 変更後は必ずテストを実行
- 変更単位の制約: 1 PR 1 目的、巨大差分を避ける
- セキュリティ制約: 秘密情報をログ・コメント・成果物に含めない

### Rule 追記例

```markdown
## 禁止事項
- 本番 DB / 本番 Cloud への write 操作禁止
- `main` ブランチへの直接 push 禁止

## 完了条件
- テストコマンドと実行結果を添付
- 変更点と影響範囲を明記
```

## 6-2. Skills（再利用できる手順の基本機能）

Skills は、Devin に渡すタスクを再利用可能な実行手順にしたものです。
定型化すると、委譲タスクの品質ばらつきを抑えられます。

### Skill 化すべき代表パターン

- バグ修正（再現、修正、テスト、PR）
- 小機能追加（仕様反映、テスト追加、ドキュメント更新）
- 依存更新（互換性確認、回帰テスト、ロールバック手順）
- PR レビュー補助（重大/中/軽微分類）

### Skill テンプレート（例）

```markdown
# Skill: バグ修正委譲

## 入力
- 再現手順
- 失敗ログ
- 対象ファイル

## 手順
1. 再現を確認
2. 原因を最小範囲で特定
3. 修正
4. テスト追加
5. PR 作成

## 完了条件
- 再現しないことを確認
- 追加/既存テストが通る
- PR に根拠を記載
```

## 6-3. Agents（役割分担の基本機能）

Devin は単体でも動きますが、役割を分けて依頼すると品質管理がしやすくなります。

### 最小 Agent 構成（推奨）

| Agent | 役割 | 禁止事項 |
| --- | --- | --- |
| 実装 Agent | 実装修正、テスト追加、PR 作成 | 本番環境への直接変更 |
| 検証 Agent | 再現確認、テスト実行、回帰確認 | 仕様変更の直接実装 |
| ドキュメント Agent | 変更概要、手順、運用記録更新 | コードの仕様変更 |

### Agent 指示の書き方

- 目的: 何を達成するか
- 入力: チケット、ログ、対象ファイル
- 完了条件: テスト結果、PR 条件、報告形式
- 禁止事項: 本番操作、機密情報出力、巨大差分

### Agent 指示テンプレート

```markdown
## Agent: 検証

目的:
- 実装後の回帰リスクを確認する

完了条件:
- 再現手順で問題が解消
- テスト結果を添付
- 未解決事項を列挙
```

## 7. 権限管理の注意事項

Devin に渡す権限は最小限にします。

| 権限 | 推奨 | 理由 |
| --- | --- | --- |
| GitHub リポジトリ（read/write） | OK（開発ブランチのみ） | PR 作成に必要 |
| 本番 DB | **NG** | 誤操作リスクが高い |
| ステージング DB | 読み取り専用まで | 検証用途に限定 |
| AWS / GCP 本番 | **NG** | 破壊的変更のリスク |
| AWS / GCP 開発 | 限定的な権限まで | サンドボックス環境 |
| Terraform apply | **NG** | 人間承認必須 |

## 8. 運用のポイント

| ポイント | 内容 |
| --- | --- |
| 完了条件を明確にする | 曖昧な指示は品質が下がる |
| 禁止事項を必ず記述する | 「本番 DB 禁止」「main 直 push 禁止」など |
| テスト方法を指定する | 「pytest tests/ が通ること」と具体的に書く |
| 権限は最小限にする | 本番・Cloud・DB の write 権限は渡さない |
| Playbook を育てる | 繰り返すタスクは Playbook 化して品質を安定させる |
| 途中経過を確認する | 長時間タスクは途中でチェックポイントを設ける |

| 業務 | Devin | 他ツール |
| --- | --- | --- |
| チケット単位の委譲 | **第一候補** | Codex（軽いタスク） |

| 技術設計・コード調査 | 向かない | **Claude Code が得意** |
| リーダー業務 | 向かない | **Codex が得意** |
